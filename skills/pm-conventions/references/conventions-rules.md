## 分析策略

### 第 1 步：清点显式文档来源（必须执行）

先扫描项目根目录及常见位置，识别**显式声明的规范文档**，这些是高置信度来源：

| 文档/配置 | 位置 | 提取的规范维度 |
|----------|------|--------------|
| `CONTRIBUTING.md` | 项目根 | Git 提交规范、分支策略、PR/审查规范、版本管理 |
| `CLAUDE.md` / `AGENTS.md` | 项目根 | 行为准则、关键约定、构建运行命令 |
| `README.md` | 项目根 | 项目级约定、关键约束 |
| `.editorconfig` | 项目根/子模块 | 缩进、行尾、字符集 |
| ESLint / Prettier / Stylelint 配置 | 前端项目 | 命名、格式、样式规范 |
| Checkstyle / SpotBugs / Spotless 配置 | Java 项目 | 命名、格式规范 |
| `.githooks/` / `.husky/` / pre-commit 配置 | 项目根 | 提交前校验规范 |

**记录纪律**：每份显式文档都要记录路径 + 它声明的规范要点（引用关键条款），不要整段复制。若项目**无任何**显式协作文档，在产出中注明"经扫描未发现显式协作文档（CONTRIBUTING/CLAUDE/AGENTS 等），以下规范全部为代码推断"。

### 第 2 步：逐维度从代码推断（7 维度全覆盖）

对以下 7 个维度逐一分析，**每个维度必须有章节**（不适用者注明"未发现/不适用"而非省略）。每个维度的"怎么推断 + 证据要求"见下。

#### 维度 1：目录规范

**怎么推断**：从 `01-项目概览.md` 的目录结构和实际 `ls/find` 结果，归纳项目的目录组织约定。

**证据要求**：给出至少 2 个模块/子项目的目录结构样例，说明共同模式（如"每个服务模块统一有 controller/service/handler 子包"）。

**置信度标准**：≥3 个模块模式一致=高；2 个一致=中；仅 1 个=低。

**典型关注点**：
- 多模块/monorepo 的模块划分与依赖方向
- 每个模块内部的子目录/子包组织（按层分 vs 按功能分）
- 源码/资源/测试的标准路径（如 Maven 的 `src/main/java`、前端的 `src/`）

#### 维度 2：命名规范（文件 / 变量 / 函数 / 类 / 组件）

**怎么推断**：抽样阅读代表性源文件，归纳命名约定。

**证据要求**：每个命名规则给出 ≥2 个具体样例（文件名/类名/标识符），最好附代码片段。命名规则要分类型陈述（Entity、Service、Controller、组件、文件、变量等）。

**置信度标准**：≥3 个同类样例一致=高；2 个=中；1 个=低。

**典型关注点**：
- 后端类命名（Entity/Mapper/Service/ServiceImpl/Controller 后缀约定）
- 数据库表/字段命名（snake_case / camelCase、主键、逻辑删除字段、审计字段）
- 前端组件命名（PascalCase 文件名）、store/API 模块命名
- 文件命名（kebab-case / camelCase / PascalCase）

#### 维度 3：代码封装规范（模块化 / 抽象层级）

**怎么推断**：结合 `02-技术栈与架构.md` 的自封装框架清单，归纳项目的封装约定。

**证据要求**：引用 `02` 文档中的自封装框架作为证据，说明项目"如何划分抽象层级"。若项目无自封装框架（极小项目），注明"经扫描未发现项目级封装框架，封装规范维度降级为记录现有分层"。

**置信度标准**：有明确自封装框架 + 多处引用=高；仅有分层无自封装=中；无明确抽象=低。

**典型关注点**：
- 分层职责边界（Controller 做什么、Service 做什么、DAO/Repository 做什么）
- 依赖注入风格（构造器注入 vs 字段注入）
- 统一报文/响应封装（是否有统一信封类）
- 跨层调用约定（是否禁止 Controller 直接调 DAO）

#### 维度 4：前端样式规范（如适用）

**适用性判定**：仅当项目有前端代码（前端模块存在 Vue/React/Angular/Svelte 文件）时执行；纯后端/CLI/library 项目**注明"不适用（无前端代码）"**。

**怎么推断**：扫描前端项目的样式相关文件（全局 CSS、设计令牌、主题配置、组件 scoped 样式）。

**证据要求**：引用具体的 CSS/样式文件路径 + 关键令牌/规则片段。若前端项目**无**自定义 CSS 方案（纯用 UI 库默认样式），注明"未发现自定义 CSS 方案/设计令牌，样式规范维度标注为'未发现'"。

**置信度标准**：有完整设计令牌系统 + 多组件遵循=高；有部分全局样式=中；仅组件内散写=低。

**典型关注点**：
- CSS 方案（全局 CSS / CSS Modules / styled-components / Tailwind / UnoCSS）
- 设计令牌系统（CSS 自定义属性 / SCSS 变量 / 主题对象）
- 组件样式作用域（scoped / shadow DOM）
- UI 库主题定制方式（覆盖 CSS 变量 vs 逐组件覆写）

#### 维度 5：错误处理

**怎么推断**：抽样阅读异常类、全局异常处理器、错误码定义、try/catch 使用模式。

**证据要求**：引用异常类/处理器/错误码的具体文件路径 + 关键代码片段（如错误码枚举值、异常映射逻辑）。

**置信度标准**：有统一异常体系 + 全局处理器 + 错误码枚举=高；有部分=中；散落 try/catch 无统一=低。

**典型关注点**：
- 统一异常类（BusinessException / DomainException 等）
- 全局异常处理器（@RestControllerAdvice / Express middleware 等）
- 错误码集中管理（枚举 / 常量类 / 配置）
- HTTP 状态码与业务错误码的映射约定（若有）
- 日志规范（@Slf4j / logger 等级使用约定）

#### 维度 6：测试规范

**怎么推断**：扫描测试目录（`src/test/`、`__tests__/`、`*.test.js`、`*.spec.ts`），统计测试文件数量与分布。

**证据要求**：给出测试文件数量、分布模块、测试框架（从依赖/导入推断）、测试类/方法命名样例。

**置信度标准（重要）**：
- 测试文件 ≥10 个且跨多模块=高，可归纳测试规范
- 测试文件 3-9 个=中，记录现有做法 + 注明"样例有限"
- 测试文件 ≤2 个=**低，禁止外推为规范**，正确表述："项目仅 N 个测试类，测试规范未成型，以下仅记录现有样例的做法（置信度低，需进一步确认）"

**典型关注点**：测试框架、测试目录约定、命名约定、Mock 风格（仅当样例充足时归纳）。

#### 维度 7：Git 提交规范（从历史/CONTRIBUTING 推断）

**怎么推断**：优先看显式文档（CONTRIBUTING.md / commitlint 配置 / .githooks）；若有 git 历史，抽样 `git log --oneline -20` 归纳 commit message 模式。

**证据要求**：
- 显式文档来源：引用 CONTRIBUTING.md 的提交规范条款
- 历史推断来源：给出 ≥10 条 commit message 样例，归纳 type/scope/格式
- 若项目**无 git 历史**（如嵌入在工作区、无独立 .git）也**无 CONTRIBUTING.md**，注明"未发现 Git 提交规范（无显式文档且无独立 git 历史）"

**置信度标准**：显式文档声明=高；历史样例 ≥10 条一致=高；历史样例 5-9 条=中；≤4 条=低。

**典型关注点**：commit message 格式（Conventional Commits / 自定义）、type 清单、分支命名、PR/审查约定、版本号策略。

### 第 3 步：边界处理与豁免

- **不适用维度**：某维度确无对应代码/文档时，**注明"未发现/不适用"并说明原因**（如"纯后端项目，无前端代码，前端样式规范不适用"），**不强制凑数、不编造**。
- **极小项目豁免**：若项目源文件总数极少（如 <20 个文件）或为单文件脚本，多数维度将降级为低置信度。在产出开头注明"项目规模极小，以下规范多为个别样例外推，置信度普遍偏低，建议人工复核"。
- **样本不足降级**：任何维度若样例不足以支撑规范（见各维度置信度标准），降级为"记录现有样例 + 低置信度标注"，**禁止外推**。
