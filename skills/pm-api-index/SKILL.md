# pm-api-index — 按模块分类、区分内外的 API 索引生成

## 定位

pm-api-index 是 project-mastery 学习管线的**波次 2** 索引 skill。它扫描项目的所有 API（对外契约 + 内部调用入口），按模块/领域分类组织，明确区分对外 API 与内部 API，为每条 API 标注精确的源码位置与调用关系，产出可供后续维护与 onboarding 快速查阅的索引文档。

与"列出所有 HTTP 路由"的区别：本 skill 覆盖**所有 API 形态**（HTTP 路由 / RPC 服务 / SPI 接口 / 前端调用层 / CLI 命令 / 事件 handler / 导出函数），不仅限于 HTTP；对每条 API 强制要求源码位置与调用关系，不是松散的清单。

## 输入

- `PROJECT_ROOT`：目标项目的根目录绝对路径
- `01-001-项目概览.md`、`02-001-技术栈与架构.md`：波次 1/2 的产出（作为上下文输入，提供项目类型、目录结构、技术栈、自封装框架位置、模块依赖关系）

## 产出

- `{PROJECT_ROOT}/docs/project-knowledge/04-001-API索引.md`

## 核心原则：每条 API 都有位置、有归属、有关系

每条 API 条目必须满足以下三条铁律，否则禁止写入产出：

1. **有位置**：每条 API（不分对外/内部）必须含 `位置: {文件路径}:{行号}` 字段。**行号基准按形态统一**：HTTP 路由指向路由注解行（如 `@*Mapping`，不是方法签名行）；Java/JS/TS/Go/Python 方法/函数指向**方法签名行**（不是注解行）；CLI 命令指向命令注册/声明行；前端调用桶指向桶对象声明行（如 `export const xxx = {`）。**内部 API 不得例外**——只给文件名不给行号不合格。
2. **有归属**：每条 API 必须归入"对外 API"或"内部 API"之一，归属判定见下节"对外 vs 内部判定方法"，不是临时分类。
3. **有关系**：至少**对外 API 与核心内部 API** 必须标注调用关系（调用方 / 被调用方 / 下游），让使用者理解该 API 在系统中的角色。零散标注不合格——必须系统覆盖对外区。

## 对外 vs 内部判定 / API 形态 / 调用关系

API 收录规则：区分对外（HTTP/RPC/GraphQL/...）vs 内部；按形态探测清单逐项执行；调用关系从 import/调用点梳理。

详细判定方法、形态探测清单、调用关系梳理、大量 API 的采样与覆盖率声明见 `references/api-rules.md`（执行时按需读取）。
## 边界处理

### 项目无明确 API 层（纯库/纯脚本/工具集）

- **纯 library**：以"导出函数（形态 7）"为对外 API，按模块/命名空间分组。
- **纯脚本/工具集**：以"CLI 命令（形态 5）"或脚本入口为对外 API。
- **确无任何 API 形态**（极简工具/配置仓库）：在产出开头注明"经探测，本项目无 HTTP/RPC/SPI/CLI/事件/导出函数等明确 API 形态，以下记录现有的可调用入口（如脚本函数、make target 等），置信度低"，**不强制凑数、不编造**。

### API 过多

按上文"采样与覆盖率声明"处理，不静默截断。

### 形态探测后发现"某形态实际是框架内置、非项目自封装"

注明"未发现（仅框架内置，非项目自封装 API）"，不算遗漏。

## 分析策略

5 步：加载上下文与形态探测 → 逐形态收录 API → 按模块/领域分类 → 调用关系梳理 → 覆盖率声明与边界检查。详细见 `references/api-rules.md`。

## 输入输出预算（小模型纪律）

- **单步单产物**：本次只产 `04-001-API索引.md`（一篇），不并行产其它。
- **读预算**：按 7 类形态探测清单逐项探测；源码打开 ≤20 个文件，单文件 ≤500 行；禁扫 node_modules/dist/target/build/.git/vendor/__pycache__；限深 ≤3 层。
- **写预算**：04 默认 8000-12000 字；对外 API top N 全标调用关系，内部 API 按被引用排序取 top N，超出部分进"待收录清单"并声明覆盖率（不静默截断）。
- **证据先行**：每条 API 含 `位置:{文件}:{行}`；行号基准按形态统一；无定位不入册。

## 执行检查清单

执行 pm-api-index 时，按以下顺序完成：

1. [ ] 读取 `01-001-项目概览.md`、`02-001-技术栈与架构.md` 作为上下文输入
2. [ ] 按"API 形态探测清单"逐项探测 7 类形态，记录发现情况
3. [ ] 逐形态提取 API 条目，每条标注【位置:文件:行】【归属:对外/内部】
4. [ ] 按判定方法区分对外/内部（注意：SPI 属于对外，不是内部）
5. [ ] 按模块/领域分类组织（不平铺），对外排前、内部排后
6. [ ] 为对外 API 全量标注调用关系（调用方/下游）；为核心内部 API 标注被调用方
7. [ ] 声明覆盖率（总数/收录数/覆盖率/采样策略）；若全量收录注明"全覆盖"
8. [ ] 处理边界（无 API 层 / 过多 API / 形态缺失）
9. [ ] 按模板格式写入 `04-001-API索引.md`
10. [ ] 自检：
    - [ ] 每条 API（含内部）都有 `文件:行`？内部 API 是否只给文件名没给行号？（不合格）
    - [ ] 对外/内部用判定方法区分？SPI 是否被正确归入对外？
    - [ ] 对外 API 是否全标了调用关系？
    - [ ] 7 类形态是否都探测过、存在的都收录了？
    - [ ] 覆盖率是否声明（含数字）？
    - [ ] 抽检 2 个 API 的 `文件:行` 在源码中可定位？
