# pm-verify — 知识库文档抽样校验（只报告不自动改）

## 定位

pm-verify 是 project-mastery 学习管线的**波次 4** 校验 skill。它读取已生成的全部 KB 文档（01-项目概览 ~ 05-构建打包部署，及 README、`_meta/`），从每份文档抽取**可验证断言**，逐条打开实际代码/配置文件对照核对，产出一份诚实的校验报告：哪些断言已验证一致、哪些不一致、哪些无法机械核对、抽样覆盖了多少。

与"读一遍文档觉得没问题"的区别：本 skill 的每一条核对都必须形成**「文档断言 → 实际代码证据（带 `{文件路径}:{行号}`）→ 比对结果」**三元组，对照实际证据、不空口确认。本 skill **只报告，不自动修改任何 KB 文档**——发现的差异写入报告的"不一致项清单"与"待人工确认可疑项清单"，由人工决定是否修订。

## 输入

- `PROJECT_ROOT`：目标项目的根目录绝对路径
- `{PROJECT_ROOT}/docs/project-knowledge/` 下已生成的全部 KB 文档（01-05，及 README、可选的 `_meta/manifest.json`）；另 `{PROJECT_ROOT}/.codebase/scan-result.json`（扫描缓存，校验类型判定时参考）

## 产出

- `{PROJECT_ROOT}/docs/project-knowledge/06-校验报告.md` — 抽样校验报告（**只产出此文件，不修改任何已生成的 KB 文档**）

## 核心原则

### 原则 1：对照实际代码证据，不空口确认（引用 verification-before-completion 理念）

本 skill 践行 superpowers `verification-before-completion` 的铁律——**「无新鲜验证证据，不得下完成结论」**。校验不是"读文档觉得合理"，而是"逐条打开代码核对"。每一条"一致/不一致"结论，背后都必须有**本次校验中实际查到的代码证据**（带 `{文件路径}:{行号}` 或具体配置字段），不能用"文档叙述流畅""章节完整""看起来对"这类**读后感式结论**替代。

本 skill 与 `verification-before-completion` 的差异：后者针对"声明任务完成前必须跑验证命令"；本 skill 把同一理念**专门用于 KB 文档 vs 代码的抽样核对**——目标是暴露文档与代码的漂移，让使用者知道"KB 里哪些已验证、哪些其实是推断、哪些已经过时"。

**禁止行为**：仅凭文档自洽性下"文档准确"的结论；只列已通过项不列未核对项；把"我没发现问题"等同于"我核对过了"。

### 原则 2：只报告，不自动修改（硬规则）

校验发现的**任何**差异（哪怕是一个标点、一个行号偏移 1），都只能写进校验报告的"不一致项清单"或"待人工确认可疑项清单"，**禁止**直接 Edit/Write 任何已生成的 KB 文档（01-05、README、`_meta/*`）。修订决策权属于人工——校验 skill 是**中立证人**，不是修订者。

**禁止行为**：发现文档写错版本号就去改文档；"顺手"修正行号偏移；把可疑项"默认按代码为准"直接覆盖文档。任何修改 KB 文档的行为都视为越权，**不合格**。

### 原则 3：每条核对产出三元组

每一条抽样核对都必须可追溯到三个字段，缺一不可：

| 字段 | 含义 |
|------|------|
| **文档断言** | 引用文档原话或转述，标注出处 `{NN-文档名}.md: {章节/行号}` |
| **代码证据** | 实际查到的代码/配置，标注 `{文件路径}:{行号}` + 关键字面量 |
| **比对结果** | 四分类之一（见下） |

### 原则 4：区分已验证与无法核对

文档里的断言分两类，核对方式不同，**结果分类也不同**：

- **可机械核对的事实断言**：版本号、端口号、入口路径、`文件:行` 标注、错误码枚举值、依赖坐标、配置字段默认值、目录结构、类/方法计数等——这些**必须实际打开代码核对**。
- **推断性 / 主观分析性陈述**：设计模式判定（"采用策略模式"）、架构原则归纳（"遵循 RESTful"）、规范外推（"项目约定 X"）、置信度自评等——这些**无法机械核对**，不应简单标"通过"，应标"无法核对（推断项）"并在置信度评估里单独说明。

## 核对结果四分类

每条核对的「比对结果」字段必须是以下四类之一：

| 分类 | 含义 | 触发条件 |
|------|------|----------|
| **一致** | 文档断言与实际代码字面量匹配 | 文档说 X，代码确为 X（允许等价表述，如版本号大小写、坐标简写） |
| **不一致** | 文档断言与实际代码冲突 | 文档说端口 8080，代码是 8081；文档说 16 个，实际 15 个；行号偏移超过容忍度（见下） |
| **无法核对（推断项）** | 断言是分析/主观判断，无机械对照基准 | 设计模式、架构原则、规范外推、置信度自评等 |
| **无法核对（断言模糊）** | 断言无明确定位（无 `文件:行`、无具体配置项），无法找到对照代码 | 文档泛泛说"使用了缓存""有异常处理"但未指位置 |

**行号容忍度**：行号偏差 ≤1 行（常见于空行/注释行计数差异）标"一致"并附注"行号偏移 ±1"；偏差 ≥2 行标"不一致"。

## 抽样策略

### 抽哪些断言（可验证性优先）

**优先核对**带以下特征的断言（这些是"机械可核对"且"高价值"的）：

1. **版本号 / 依赖坐标**：`pom.xml`/`package.json`/`go.mod` 等里的版本声明
2. **`文件:行` 标注**：文档里出现的所有 `{文件路径}:{行号}`，逐条核对那一行确实是文档声称的内容
3. **配置值**：端口号、环境变量名、默认值、占位符（`${VAR:default}`）
4. **入口点 / 关键路径**：启动类、入口文件、模块路径
5. **枚举 / 计数**：错误码总数、API 端点数、模块数、Service/Controller 计数
6. **目录结构声明**：文档画的目录树是否与实际 `ls`/`find` 结果一致
7. **关键代码片段**：文档引用的代码片段（类名、方法签名、注解）是否在源码中真实存在

**降级处理**推断性陈述（设计模式、原则归纳）——标"无法核对（推断项）"，不强行下"通过/不一致"结论。

### 每份文档抽多少

- **每份 KB 文档至少核对 ≥3 条断言**（少于 3 条不足以代表该文档）。
- **覆盖该文档内不同类型的断言**——不能只挑版本号一类。例如 02-技术栈文档应同时覆盖版本号、自封装框架位置、模块依赖方向；04-API 索引应覆盖端点行号、归属判定、调用关系三类。
- **优先核对"高价值 + 易漂移"的断言**——跨文档重复出现的数字（如某计数在 02、04、README 都出现）、`文件:行` 标注（行号最易随代码演进漂移）。

### 抽样范围声明（强制）

无论核对多少，都必须在报告里**量化声明抽样范围**——这是诚实性的底线，让使用者知道"通过"结论的可信边界：

- 核对了哪几份文档
- 每份核对了多少条断言
- 用了什么抽样方法（可验证性优先 / 重点类型覆盖）
- 未抽样的断言如何处理（**不再外推为"全部通过"**，标"未抽样，不下整体结论"）

**即便全过**，也必须声明抽样范围——"全部通过"是基于核对 N 条得出的，不是基于核对 100% 断言。

## 对照核对方法

### 第 1 步：清点 KB 文档与可验证断言

1. `ls` `{PROJECT_ROOT}/docs/project-knowledge/`，列出全部已生成的 KB 文档
2. 逐份阅读，**抽取可验证断言**——按"抽样策略"优先级挑，记录每条的出处（`{NN-文档名}.md: {章节}`）
3. 区分**可机械核对** vs **推断性陈述**，后者直接标"无法核对（推断项）"不进入对照流程

### 第 2 步：逐条对照代码（必须执行）

对每条可机械核对的断言：

1. **定位代码**：根据断言里的 `文件:行` / 配置字段 / 类名，用 `cat`/`grep -n`/`sed -n`/Read 打开实际文件到具体行
2. **提取字面量**：读到那一行的实际内容（不是凭印象）
3. **比对**：按"核对结果四分类"判定
4. **记录三元组**：文档断言 + 代码证据（带 `{文件路径}:{行号}`）+ 比对结果

**关键纪律**：必须真的打开文件读到那一行，不能"文档里写了文件:行，看起来合理"就标一致——那是橡皮图章。

### 第 3 步：「未发现」声明独立复核

KB 文档里有大量"未发现 X"类声明（如"未发现 Dockerfile""未发现 CI 配置""未发现测试目录"）。这类声明容易**虚报**（生成时漏探测）或**过时**（生成后项目新增了）。对每条"未发现"声明**独立重新探测**一次，确认确实不存在。

### 第 4 步：跨文档一致性检查

同一事实在多份文档里重复出现时（典型：模块数、API 总数、错误码数在 02/04/README 都出现），核对**文档之间是否自洽**。文档内部或文档之间自相矛盾（如 04 说 16 个、README 说 15 个）单列为不一致项，即便其中某个数字与代码一致。

### 第 5 步：置信度评估

对每份 KB 文档给出**校验后置信度评估**，纳入以下因素：

| 因素 | 对置信度的影响 |
|------|---------------|
| 抽样核对中"一致"项占比高 | 上调 |
| 发现"不一致"项 | 下调（按严重度） |
| 文档自评置信度与本次核对结果吻合 | 维持/微调 |
| 推断性陈述占比高（大量"无法核对（推断项）"） | 下调（说明文档里"事实"与"推断"混杂） |
| "未发现"声明经复核准确 | 维持 |

置信度等级沿用 pm-scan 的 high/medium/low 三级，但**必须基于本次抽样核对的实际结果**给，不能照抄文档自评。

## 边界处理

### 文档断言无法核对（推断项为主）

当某份文档（或某章节）大量是推断性陈述（典型：02 的设计模式分析、03 的规范外推），**不要强行下"通过"结论**。正确做法：在抽样核对里标"无法核对（推断项）"，在置信度评估里注明"本文档推断性内容占比高，机械可核对的事实断言核对结果为 X/Y，其余无法核对"。

### 文档断言模糊无定位

文档泛泛说"使用了缓存""有统一异常处理"但未给 `文件:行` 或具体配置——标"无法核对（断言模糊）"，并在报告里建议"该断言应补充具体位置以便后续核对"。**不编造对照位置**。

### KB 文档不完整（部分未生成）

若 `docs/project-knowledge/` 下只有部分 KB 文档（如只有 01-03，04/05 未生成），**只校验已生成的**，在报告里显式声明"未生成的文档（04/05）未纳入本次校验"。

### 项目源码与文档生成时相比已演进

若发现大量行号偏差、类名不存在（项目源码已演进，文档生成时的代码已变），在报告开头注明"检测到源码可能已演进，行号/类名偏差较多，建议重新执行波次 1-3 生成 skill 后再校验"。**不**自行修订文档，只报告。

### KB 文档之间相互引用失同步

如 README 引用了 04 的统计数字，但 04 后来更新了数字而 README 没跟上——单列为"跨文档不一致"项。

## 八、校验方法说明

本报告通过以下方式交叉核对：
1. **配置文件核对**：{列出本次实际读过的配置文件}
2. **源码行号核对**：用 `grep -n`/Read 逐条比对 KB 声明的 `{文件路径}:{行号}` 与源码实际行号
