# lp-feature-scan — 项目可学习功能识别

## 定位

lp-feature-scan 是 learn-project（项目学习文档生成）管线的**第 1 步**功能扫描 skill。它扫描一个已有项目，识别"值得学习"的功能/能力单元，产出一份**项目特定的、可勾选的功能清单**，供后续人在回路挑选、再由 lp-prompt-gen 深挖生成学习文档。

与"列目录树"或"列业界通用功能"的区别：本 skill 的每一条功能都必须是**该项目特有的、可被学习者当作一个整体来理解的能力单元**（如某个核心业务能力、某套自封装框架、某个特色机制），且每条附核心度/复杂度/依赖/证据四字段，能驱动排序与深挖。

## 输入

- `PROJECT_ROOT`：目标项目的根目录绝对路径
- `{PROJECT_ROOT}/docs/project-knowledge/01-项目概览.md`、`02-技术栈与架构.md`：定位阶段（pm-scan/pm-techstack）的产出，作为上下文输入（提供项目类型、目录结构、技术栈、自封装框架位置）。若不存在，本 skill 自行做轻量顶层扫描。

## 产出

- `{PROJECT_ROOT}/docs/learning-docs/features/inventory.md` — 功能清单（含未选中的，按核心度排序，每条带 `- [ ]` 勾选框）

## 核心原则

1. **项目特定，非通用模板**：功能必须是该项目语境下真实存在、值得专门学习的能力。禁止输出"用户登录""CRUD""权限管理"这类放之四海而皆准的通用清单——除非该项目在该能力上有独特设计值得学习（此时名称要体现独特性，证据要指向独特实现）。
2. **粒度适中**：一个功能 = 一个可被学习者整体理解的能力单元。过细（每端点/每文件一个）或过粗（"整个后端"）都要避免。相近的小能力应聚类（如一组相关 CRUD 端点归为"X 资源管理"）。
3. **四字段齐全**：每条功能必须含核心度、复杂度、依赖、证据，缺一不可——它们驱动排序与后续 lp-prompt-gen 深挖。
4. **证据真实可验证**：证据必须是指向真实存在文件/目录/入口/文档的 `{路径}: {说明}`，禁止"相关代码""核心逻辑"等模糊描述。
5. **零臆造**：不发明项目里不存在的功能或路径。

## 扫描方法论：5 层扫描（先顶层后深扫，不盲目遍历）

沿用"先拿足够信息再决定是否深扫"的纪律，按下述 5 层依次进行，最后聚类去重。

### 第 1 层：文档层（最权威的功能框架）

先读项目自己如何描述自己——这是识别功能最权威的来源：

| 来源 | 位置 | 提取什么 |
|------|------|----------|
| `README.md` | 项目根 | 项目自我宣传的核心能力、特性列表、用例 |
| `docs/` 目录 | 项目根/docs | 设计文档、特性文档、模块说明的目录大纲 |
| `CHANGELOG.md` / `FEATURES.md` | 项目根 | 历史迭代沉淀出的功能项 |

**记录**：把文档里明示的功能/主题记为候选功能，附文档路径作为证据来源。

若项目**无任何说明文档**，本层注明"未发现说明文档，功能识别依赖结构层/入口层推断"。

### 第 2 层：结构层（候选功能区）

从顶层模块/包/目录识别候选功能区：

1. `ls` 顶层目录，跳过无关目录（`node_modules/`、`target/`、`build/`、`.git/`、`dist/`、`vendor/` 等）。
2. 对多模块/monorepo 项目，每个顶层模块/子项目通常是一个功能区候选。
3. 对单模块项目，看内部的按功能划分的子包/子目录（按功能分 vs 按层分——按功能分的子目录更可能是功能单元）。

**记录**：每个候选功能区附 `{目录路径}: {该目录承担什么职责}`。

### 第 3 层：入口层（用户可感知的能力）

定位用户/外部可感知的入口，反推能力：

| 入口类型 | 探测方式 | 提取的能力 |
|---------|----------|-----------|
| CLI 命令 | `package.json` 的 `bin`、`*.cli.*`、`cmd/*/main.go`、argparse/cobra/commander | 每个命令对应一项能力 |
| API 路由组 | 路由定义（`@*Mapping`、`router.*`、`app.route`、`@Controller` 等）按资源/前缀分组 | 每组资源对应一项能力 |
| 导出符号 | 库的 `index`/`main` 导出、`__all__`、public API | 每组导出对应一项能力 |
| 插件/扩展注册点 | 插件目录、注册函数、SPI/扩展点配置 | 每类扩展点对应一项能力 |

**记录**：每项能力附 `{入口文件}: {说明，如"定义了 X 资源的增删改查端点"}`。

### 第 4 层：示例层（展示的用例）

扫描示例/演示，反推被展示的能力：

- `examples/`、`demo/`、`samples/`、`docs/examples/` 下的每个示例通常演示一项能力。

**记录**：作为第 1-3 层的补充证据或新候选。

### 第 5 层：聚类去重

交叉比对前 4 层的候选，合并成统一功能清单：

- 同一能力在多层出现 → 合并为一条，证据合并。
- 粒度不一致 → 统一到"一个可整体理解的能力单元"。
- 去掉纯技术实现细节（如"用了 Redis 缓存"不是功能，是某功能的技术手段）。

## 功能清单字段

每条功能必须含：

| 字段 | 取值 | 用途 |
|------|------|------|
| 名称 | 项目语境下的自然名称 | 标识 |
| 简介 | 一句话：它做什么 | 让挑选者快速判断 |
| 核心度 | `核心` / `边缘` | 是否属于项目主线 → 驱动排序 |
| 复杂度 | `高` / `中` / `低` | 学习成本 → 驱动路径排序 |
| 依赖 | 学它前建议先懂的功能/概念；无则标 `无` | 驱动学习路径前后顺序 |
| 证据 | `{路径}: {说明}`（可多条） | 供 lp-prompt-gen 深挖、可验证真实性 |

## 大项目取舍

- **限制扫描深度**：同 pm-scan 纪律，一般扫描不超过 3 层目录深度，读取配置/入口文件总数不超过 20 个。
- **按核心度排序**：产出按核心度排序（核心在前），不强求穷举所有功能。
- **交人挑选**：清单可能很长，依赖人在回路挑选要深入的功能——本 skill 只负责把候选呈现清楚，不替用户决定学哪些。

## 边界处理

- **无说明文档**：第 1 层注明"未发现说明文档"，从结构层/入口层推断，并在清单开头标注"功能识别基于结构与入口推断"。
- **功能粒度过细**：相近的小能力聚类成一个（如一组 CRUD 端点归为"X 资源管理"），不在清单里塞几十条碎功能。
- **极小项目**：功能数少则如实呈现（可能 1-3 个），不凑数、不把单个文件拆成多个"功能"。
- **无单一功能入口（纯库/纯聚合）**：依赖导出符号层（第 3 层）与文档层识别，注明"无运行时入口，功能以导出 API 形式存在"。
- **核心度难判定**：当难以区分核心/边缘时，结合文档层是否重点宣传 + 入口层是否高频出现综合判断，标注理由。

## 执行检查清单

执行 lp-feature-scan 时，按以下顺序完成：

1. [ ] 读取 `01-项目概览.md`、`02-技术栈与架构.md`（若存在）作为上下文
2. [ ] 第 1 层：读 README/docs/CHANGELOG，提取文档明示的功能候选
3. [ ] 第 2 层：`ls` 顶层结构，识别候选功能区
4. [ ] 第 3 层：定位 CLI/API/导出/插件入口，反推用户可感知能力
5. [ ] 第 4 层：扫 examples/demo，补充证据
6. [ ] 第 5 层：聚类去重，统一粒度
7. [ ] 为每条功能填四字段（核心度/复杂度/依赖/证据），证据为真实路径+说明
8. [ ] 按核心度排序，每条加 `- [ ]` 勾选框
9. [ ] 写入 `features/inventory.md`
10. [ ] 自检：每条是项目特定功能？四字段齐全？证据真实可验证？零臆造？勾选框在？按核心度排序？
