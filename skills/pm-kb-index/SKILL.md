# pm-kb-index — 知识库总索引与任务型阅读路径汇编

## 定位

pm-kb-index 是 project-mastery 学习管线的**波次 3** 索引 skill。它读取已生成的全部 KB 文档（01-项目概览 ~ 05-构建打包部署，及已有的 06-校验报告），汇编一份**知识库前门 README**：让读者在 30 秒内知道"这是什么项目、KB 里有哪些文档、我要做什么任务时该读哪几份"。

**由主会话执行，非 agent**——本 skill 不做源码分析，只汇编已有 KB 文档的索引。执行者读全部已生成文档 + `_meta/` 元信息，产出 README；不 dispatch subagent 做 GREEN 验证（baseline RED 可 dispatch）。

与"列一份文档清单"的区别：本 skill 的产出是**任务驱动的查阅入口**，不是目录树。它强制要求项目一句话简介、每份文档的"何时读"触发式描述、≥2 条基于任务的阅读路径、从 `_meta/` 读取的元信息、对未生成文档的显式标注。

## 输入

- `PROJECT_ROOT`：目标项目的根目录绝对路径
- `{PROJECT_ROOT}/docs/project-knowledge/` 下已生成的全部 KB 文档（01-项目概览 ~ 05-构建打包部署，06-校验报告可能尚未生成）
- `{PROJECT_ROOT}/.codebase/scan-result.json`：pm-scan 产的机器契约（项目名/类型/扫描时间/skill 版本，元信息主来源）
- `{PROJECT_ROOT}/docs/project-knowledge/_meta/manifest.json`：收尾写的完成态记录（sourceCommit/skillVersions，若已收尾）

## 产出

- `{PROJECT_ROOT}/docs/project-knowledge/README.md` — 知识库前门索引

## 核心原则：README 是知识库前门，不是目录树

README 必须满足以下五条铁律，否则禁止定稿：

1. **有项目一句话简介**：README 开篇（在文档导航之前）必须有一句话让读者建立"这是什么项目"的整体认知。这一句话从 01-项目概览 提炼，不是复制标题。定位是"前门"而非"目录"。
2. **每份文档有「何时读」**：文档导航中**每份文档**必须有触发式的"何时读"描述（"当你想 X 时读这份 / 想做 Y 时先读这份"），不是仅"这份文档讲什么"的内容陈述。让读者看一眼就知道要不要读。
3. **有任务型阅读路径**：README 必须含"基于任务的阅读路径"章节，**≥2 条**，每条标注任务名 + 阅读顺序（先读什么、再读什么、跳读哪些章节）。覆盖项目最典型的工作流：开发新功能 / 部署 / 改 API 至少各一条；推荐补充：排查 bug / 改核心引擎或核心逻辑 / 接手项目快速了解。
4. **有元信息且从缓存/manifest 读**：README 必须含元信息章节（生成时间、源码版本/commit、KB 文档清单与完成状态、skill 版本）。元信息字段从 `.codebase/scan-result.json`（`project.name` / `generated_at` / `scanner.version` / `classifications`）与 `_meta/manifest.json`（若存在，`sourceCommit` / `skillVersions`）读取；**字段无法读取时显式标注"未生成/未记录"，禁止编造当前时间、禁止跳过该章节**。
5. **区分已生成与未生成文档**：对规划中但尚未生成的文档（典型：06-校验报告，波次 3 未跑），在导航中**显式标注"未生成"**，不静默忽略、不给死链。让使用者知道 KB 的完整规划与当前完成度。

**禁止行为**：把 README 写成纯目录树（只有文件名/标题）；编造生成时间而不查 `_meta/`；只列已生成文档让使用者误以为 KB 完整；文档导航只写"这份讲什么"不写"何时读"。

## 元信息读取规则

元信息章节的字段来源（按存在性探测，缺失则标注）：

| 字段 | 来源（优先级从高到低） | 缺失时 |
|------|----------------------|--------|
| 项目名称 | `.codebase/scan-result.json` 的 `project.name`；若不存在则从 01-项目概览 提炼 | 标注"未记录" |
| 生成时间（KB 起始扫描时间） | `.codebase/scan-result.json` 的 `generated_at`（pm-scan 时间戳，代表 KB 生成的起始时刻）；若存在 `_meta/manifest.json` 的 `generatedAt` 则优先用后者（KB 完成时刻） | 标注"未记录（无时间戳）"，**禁止编造当前时间** |
| skill 版本 | `.codebase/scan-result.json` 的 `scanner.version`（pm-scan）；`_meta/manifest.json` 的 `skillVersions`（若存在，合并各 skill 版本） | 标注"未记录" |
| 项目类型判定 | `.codebase/scan-result.json` 的 `classifications`（取 `is_primary=true` 的 type，及全量 type 列表 + confidence） | 标注"未记录" |
| 源码版本/commit | `_meta/manifest.json` 的 `sourceCommit` / `sourceVersion`（若存在）；否则尝试 `git -C {PROJECT_ROOT} rev-parse --short HEAD`（若有 .git） | 都无则标注"未记录（无 git 仓库或 _meta 无 commit 字段）" |
| KB 文档清单与完成状态 | 实际 `ls` `{PROJECT_ROOT}/docs/project-knowledge/` 的结果，对照规划清单（01-06）标注已生成/未生成 | — |

**关键规则**：`_meta/manifest.json` 是可选的（收尾才写）——pm-scan 产 `.codebase/scan-result.json`，`manifest.json` 由收尾步骤补充。本 skill 不得因 `manifest.json` 不存在而报错或跳过元信息章节，必须从 `.codebase/scan-result.json` 读已有字段，其余显式标注"未生成/未记录"。

## 文档导航结构规则

文档导航章节（"知识库文档地图"或类似标题）必须对**每份已生成的 KB 文档**给出：

| 导航要素 | 要求 |
|---------|------|
| 文档名 + 相对链接 | 用 Markdown 链接（`[01-项目概览.md](01-项目概览.md)`），相对路径，可在仓库内点击跳转 |
| 「何时读」一句话 | **触发式表述**："当你想 X 时读这份 / 想做 Y 时先读这份"。每份文档的触发条件应互不重叠（让读者按意图分流） |
| 一句话内容定位 | 简述这份文档覆盖什么（补充"何时读"，不替代它） |

**各 KB 文档的「何时读」参考触发条件**（通用，跨项目适用）：

| KB 文档 | 典型「何时读」触发条件（按项目实际调整措辞） |
|---------|--------------------------------------------|
| 01-项目概览 | 想快速了解"这是什么项目、多大、入口在哪"时先读；任何任务的起点 |
| 02-技术栈与架构 | 想知道"用了哪些框架、怎么分层、数据怎么流"时读；改架构、加自封装框架前必读 |
| 03-开发规范 | 想写代码、改代码、提交 PR 前必读；想知道"目录怎么放、类怎么命名、异常怎么抛、提交怎么写"时读 |
| 04-API索引 | 想改/加一个 API、排查 API 报错、找调用关系时读；想知道"有哪些 API、在哪个文件第几行"时读 |
| 05-构建打包部署 | 想把项目跑起来、打包、部署、配环境变量时读；想知道"怎么编、怎么打、怎么跑"时读 |
| 06-校验报告 | 想知道"KB 文档的抽样校验结果、哪些章节置信度低需人工复核"时读（若已生成） |

## 任务型阅读路径设计规则

"基于任务的阅读路径"章节是 README 的**核心价值章节**。设计要求：

1. **路径数 ≥2 条**，推荐 4-6 条覆盖项目最典型工作流。
2. **必须覆盖的工作流**（至少各一条）：
   - **开发新功能 / 改代码**：先读规范（03）确认放哪、怎么命名 → 再读架构（02）了解分层 → 再读 API 索引（04）找现有入口
   - **部署 / 上生产**：读构建打包部署（05）
   - **改 API / 排查 API 报错**：读 API 索引（04）+ 相关架构（02）
3. **推荐补充工作流**（按项目类型选用）：
   - **接手项目快速了解**：01 → 02 的架构/数据流 → 跑起来（05）
   - **改核心引擎 / 核心逻辑**：02 的核心组件 → 04 的调用关系 → 03 的红线
   - **排查 bug**：04 按路径找端点 → 看下游 → 看错误码（03 的错误处理）
4. **每条路径必须标注**：任务名（"我要 X"的第一人称表述）+ 阅读顺序（先读哪份的哪节、再读哪份）+ 关键提示（红线、注意事项）。
5. **路径要具体到"读哪份的哪一节"**，不是泛泛"读 03"。例如："读 03 的'目录规范''命名规范''错误处理'三节"。

## 边界处理

### KB 文档不完整（部分文档未生成）

- 规划中的文档（如 06-校验报告）未生成时，在文档导航中**显式列出并标注"未生成"**（如"06-校验报告.md — 未生成（波次 3 校验尚未执行）"），**不提供链接或链接前注明"待生成"**。
- 在元信息章节的"KB 文档清单与完成状态"中标注完成度（如"已生成 5/6，06-校验报告待生成"）。
- 禁止静默忽略未生成文档——使用者必须知道 KB 的完整规划与当前缺口。

### `_meta/` 不完整

- `_meta/manifest.json` 不存在：从 `.codebase/scan-result.json` 读已有字段（`project.name`/`generated_at`/`scanner.version`/`classifications`），其余字段显式标注"未记录"。**禁止因 manifest.json 不存在跳过元信息章节**。
- `.codebase/scan-result.json` 也不存在（pm-scan 未跑或被删）：元信息章节大部分字段标注"未记录（.codebase/scan-result.json 不存在，pm-scan 可能未执行）"，并在 README 开头提示"KB 元信息缺失，建议先执行 pm-scan 生成 .codebase/scan-result.json"。
- 禁止编造 `_meta/` 中不存在的字段值（特别是生成时间）。

### KB 目录为空或仅有 README

- 若 `docs/project-knowledge/` 下没有任何 KB 文档（01-05 均无），本 skill **不应执行**——没有内容可索引。在产出中注明"未检测到已生成的 KB 文档，请先执行波次 1/2 的各 skill"，不强制汇编空 README。

### 项目无 git 仓库

- 源码版本/commit 字段：尝试 `git rev-parse` 失败时，标注"未记录（项目无 git 仓库）"，不报错。
