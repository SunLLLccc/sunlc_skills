# pm-verify-lite 抽样规则与升级触发（按需读取）

> 由 pm-verify-lite SKILL.md 抽出，执行时按需加载。

## 抽样优先级（可验证性优先）

每份 KB 文档抽样 **3-5 条**断言（封顶）。优先抽"机械可核对 + 高价值 + 易漂移"的：

| 优先级 | 断言类型 | 为什么优先 |
|--------|----------|-----------|
| ★★★ | `文件:行` 标注 | 行号最易随代码演进漂移，且可机械定位 |
| ★★★ | 版本号 / 依赖坐标 | 配置文件里有字面量，机械可比 |
| ★★ | 配置值（端口/环境变量/默认值/占位符） | 配置事实，机械可比 |
| ★★ | 枚举 / 计数（API 数/模块数/错误码数） | `grep -c`/`find` 可复核 |
| ★ | 入口点 / 关键路径 | 启动类/入口文件可定位 |
| ★ | 目录结构声明 | `ls`/`find` 可复核 |
| 降级 | 推断性陈述（设计模式/架构原则/规范外推） | 无机械基准，标"无法核对" |

**覆盖不同类型**：3-5 条不要全挑版本号一类。例：02-技术栈文档应同时覆盖版本号、自封装框架位置、模块依赖方向；04-API 索引应覆盖端点行号、归属判定、调用关系。

## lite 与 full 的核心差异（执行时参照）

| 维度 | lite（本 skill，默认） | full（pm-verify，opt-in） |
|------|------------------------|---------------------------|
| 每文档抽样 | 3-5 条封顶 | ≥3 条，覆盖全部断言类型 |
| 比对分类 | 三分类 | 四分类（无法核对细分推断项/断言模糊） |
| 跨文档一致性 | 不做 | 做 |
| 「未发现」独立复核 | 不做 | 逐条重新探测 |
| 逐文档置信度 | 不做（整体风险信号） | 做（high/medium/low） |
| 输出 | ≤5000 字 | 全量 |

## 升级触发条件（lite → full）

lite 报告"风险信号"为以下任一时，建议改跑 full `pm-verify`：

1. **不一致项 ≥3 条**——抽样已暴露明显漂移，需全量定位范围。
2. **任一关键文档（01-001-项目概览 / 02-001-技术栈与架构 / 04-001-API索引）出现不一致**——这三份是后续维护/onboarding 的高频依据，错一个误导大。
3. **某文档"无法核对"占比 ≥60%**——说明该文档推断性内容过浓，需 full 的四分类 + 置信度评估厘清事实/推断边界。
4. **人工指定 / 关键项目**——直接跑 full，不经 lite。

## lite 的"不做"边界（诚实声明）

lite **不做**以下 full 版职责，并在报告末尾标注"如需深度核对请跑 pm-verify"：

- **跨文档一致性**：同一事实（如模块数、API 总数）在 02/04/README 重复出现时的自洽核对——lite 不做。
- **「未发现」独立复核**：KB 文档里"未发现 Dockerfile/CI/测试目录"类声明，full 会逐条重新探测确认确不存在——lite 不做（仅在抽样中偶然碰到才记录）。
- **逐文档置信度评估**：full 给每份文档 high/medium/low——lite 只给整体一句风险信号。

这些"不做"是 lite 控制单次输出（≤5000 字）的必要取舍，不是遗漏——需时升级 full。
