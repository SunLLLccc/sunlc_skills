# pm-verify-lite — 知识库文档轻量抽样校验（默认，只报告不修改）

## 定位

pm-verify-lite 是 project-mastery 学习管线的**波次 4 默认校验 skill**。它对每份已生成的 KB 文档**抽样 3-5 条**可机械核对的断言，逐条打开实际代码对照，产出一份**短报告**（默认 ≤5000 字）：哪些一致、哪些不一致、哪些无法核对、整体风险信号。

与完整 `pm-verify` 的区别：

| 维度 | pm-verify-lite（默认） | pm-verify（opt-in） |
|------|------------------------|---------------------|
| 每文档抽样 | 3-5 条（封顶） | ≥3 条，覆盖全部断言类型 |
| 比对结果分类 | 三分类（一致/不一致/无法核对） | 四分类（+ 无法核对细分推断项/断言模糊） |
| 跨文档一致性 | 不做 | 做（同一事实多文档自洽核对） |
| 「未发现」独立复核 | 不做 | 逐条重新探测 |
| 置信度评估 | 整体一句风险信号 | 逐文档 high/medium/low |
| 输出长度 | ≤5000 字 | 无硬上限（全量） |
| 触发 | 默认（波次 4） | 人工指定 / 关键项目 / lite 报高风险 |

**何时升级到 full**：lite 报告的"风险信号"为高风险（不一致项 ≥3 条，或任一关键文档——01/02/04——出现不一致），编排器/人工应改跑完整 `pm-verify`。

**铁律**：只报告，不自动修改任何 KB 文档（同 pm-verify）。每条核对必须形成**「文档断言 → 代码证据（`{文件路径}:{行号}`）→ 比对结果」**三元组，对照实际证据、不空口确认。

## 输入

- `PROJECT_ROOT`：目标项目的根目录绝对路径
- `{PROJECT_ROOT}/docs/project-knowledge/` 下已生成的全部 KB 文档（01-05，及 README）
- `{PROJECT_ROOT}/.codebase/scan-result.json`（扫描缓存，核对类型判定时参考）

## 产出

- `{PROJECT_ROOT}/docs/project-knowledge/06-001-校验报告.md` — 轻量抽样校验报告（**只产出此文件，不修改任何已生成的 KB 文档**；文件名与 full 版一致，内容更短）

## 输入输出预算（小模型纪律）

- **单步单产物**：本次只产 `06-001-校验报告.md`（一份），不并行产其它。
- **读预算**：读全部已生成 KB 文档（01-05/README）+ scan-result.json；抽样核对时每条用 `grep -n`/Read 打开实际代码到具体行，单文件 ≤500 行；禁扫 node_modules/dist/target/build/.git/vendor/__pycache__。
- **写预算**：报告默认 ≤5000 字；每文档抽样 3-5 条（封顶），超出预算的核对留给 full 版。
- **证据先行**：每条核对带 `{文件路径}:{行号}` 代码证据；无证据的推断项标"无法核对"，不空口确认。

## 核心原则

1. **对照实际代码证据**：每条"一致/不一致"结论背后必须有本次实际查到的代码证据（`{文件路径}:{行号}` + 关键字面量），不能用"叙述流畅""看起来对"替代。
2. **只报告不修改**（硬规则）：发现的差异只写进报告，**禁止** Edit/Write 任何已生成的 KB 文档（01-05/README/_meta）。修订决策权属人工。
3. **可机械核对优先**：优先抽版本号/`文件:行`/配置值/入口路径/枚举计数/目录结构——这些有明确机械基准；推断性陈述（设计模式、架构原则）标"无法核对"不强行下结论。
4. **诚实边界**：报告必须量化声明抽样范围（核了几份/几条），即便全过也不外推为"100% 正确"。

## 比对结果三分类

每条核对的「比对结果」字段是以下三类之一（lite 版合并了 full 的两个"无法核对"子类）：

| 分类 | 含义 |
|------|------|
| **一致** | 文档断言与实际代码字面量匹配（行号偏差 ≤1 视为一致并附注） |
| **不一致** | 文档断言与实际代码冲突（版本号/端口/计数/行号偏差 ≥2 等） |
| **无法核对** | 推断性陈述无机械基准，或断言模糊无定位——不强行下结论 |

## 抽样策略

每份已生成 KB 文档抽样 **3-5 条**断言核对（封顶），优先可机械核对的：

1. 版本号 / 依赖坐标（pom.xml/package.json/go.mod 等）
2. `文件:行` 标注（逐条核对该行确实是文档声称的内容）
3. 配置值（端口/环境变量/默认值/占位符）
4. 入口点 / 关键路径
5. 枚举 / 计数（API 数、模块数、错误码数）
6. 目录结构声明

降级处理推断性陈述——标"无法核对"。详细抽样优先级、lite 与 full 的对照、升级触发条件见 `references/verify-lite-rules.md`（执行时按需读取）。

## 边界处理

- **KB 文档不完整**：只校验已生成的，报告开头声明"未生成的文档（如 04/05）未纳入本次校验"。
- **大量推断性内容**（02 设计模式/03 规范外推）：如实标"无法核对"占比高，风险信号注明"本文档推断性内容占比高，机械核对覆盖有限"。
- **源码已演进**（行号/类名偏差多）：报告开头注明"检测到源码可能已演进，建议重跑波次 1-3 后再校验"，只报告不修订。
- **lite 不做的事**：跨文档一致性核对、「未发现」声明独立复核、逐文档置信度评估——这些是 full 版职责，lite 在报告末尾标注"如需深度核对请跑 pm-verify（opt-in）"。

## 执行检查清单

执行 pm-verify-lite 时，按以下顺序完成：

1. [ ] `ls` `{PROJECT_ROOT}/docs/project-knowledge/`，清点全部已生成的 KB 文档
2. [ ] 逐份阅读 KB 文档，**每份抽 3-5 条**可机械核对的断言（优先版本号/文件:行/配置值/计数/目录）
3. [ ] 逐条打开实际代码核对（用 `grep -n`/Read 真的读到那一行，不是凭印象）
4. [ ] 每条核对产出三元组：文档断言 + 代码证据（`{文件路径}:{行号}`）+ 三分类结果
5. [ ] 推断性陈述标"无法核对"，不强行下结论
6. [ ] 汇总：一致/不一致/无法核对 各几条；整体风险信号（低/中/高）；是否建议升级 full
7. [ ] 按模板格式写入 `06-001-校验报告.md`（≤5000 字）
8. [ ] 自检：
    - [ ] **是否修改了任何已生成的 KB 文档？（不合格——硬规则违反）**
    - [ ] 每条核对有三元组？代码证据带 `{文件路径}:{行号}`？还是只有读后感？（不合格）
    - [ ] 每份文档抽样 3-5 条？覆盖不同类型断言？
    - [ ] 抽样范围量化声明（几份/几条）？即便全过也声明了边界？
    - [ ] 不一致项显式列出（哪怕 1 条）？
    - [ ] 报告 ≤5000 字？是否给出风险信号与升级建议？
