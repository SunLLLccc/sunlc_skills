# pm-techstack-generic 压力场景

## 场景描述

让一个 general-purpose subagent（不带任何 skill）分析 dsp 项目的技术栈与架构，然后对比其产出与期望质量的差距。

## 期望跑偏点（baseline 失败模式）

1. **只罗列依赖**：直接读 pom.xml 和 package.json，把依赖名罗列一遍就完事——不分析用途、不找配置文件位置
2. **漏掉项目自封装框架**：dsp 有 dsp-common（ApiRequest/ApiResponse/ErrorCode/JwtUtil/DataQueryService 接口）、dsp-core（CRUD 基类层）、dsp-engine（XML 执行引擎，含 4 种执行器/编排器/29 个内置函数）等明显的自封装模块——baseline 很可能只把它们当普通子模块一笔带过，不识别为"项目自造的内部框架"
3. **不区分开源 vs 自造**：把所有依赖混在一起列，不标注哪些是开源第三方、哪些是项目自封装
4. **版本缺失**：不记录每个框架的版本号
5. **架构模式缺失**：不分析分层架构、模块划分、数据流
6. **模块依赖关系缺失**：不画/不列模块间的依赖关系图
7. **无产出模板**：输出格式随意，不符合 `02-技术栈与架构.md` 的结构化模板

## 验证指标

跑 baseline 后，检查产出是否包含：
- [ ] 独立的「自封装框架」章节
- [ ] 自封装框架条目 ≥1（如 dsp-common、dsp-core、dsp-engine）
- [ ] 开源框架条目含版本号
- [ ] 开源框架条目含用途说明
- [ ] 开源框架条目含配置文件位置
- [ ] 架构模式分析（分层/模块划分）
- [ ] 模块依赖关系
- [ ] 符合 02-技术栈与架构.md 模板结构
