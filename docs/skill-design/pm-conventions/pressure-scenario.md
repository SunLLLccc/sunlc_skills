# pm-conventions 压力场景

## Baseline 失败模式

不加 pm-conventions skill，让模型直接"提取 dsp 项目的开发规范"时，预期出现以下 5 类跑偏：

### 失败模式 1：臆造通用最佳实践，不对照项目实际代码

模型直接输出"使用 RESTful 命名"、"Controller 层只做参数校验"、"Service 层必须写单元测试"等通用最佳实践——这些规范可能和项目实际做法完全不一致，但没有去代码中验证。

**验证方式**：检查每条规范是否附有具体的代码证据（文件路径 + 代码片段/模式），还是仅凭"业界惯例"列出。

**GREEN 规则**：SKILL"核心原则"要求"每条规范必须能在代码中找到证据"，产出模板每条规范必须包含"证据"和"推断置信度"字段，无证据的规范禁止写入。

### 失败模式 2：规范与项目真实用法不符

模型猜测的规范和项目实际写法矛盾。例如实际项目用 `ApiRequest<T>` 统一信封但模型写"接口直接返回 POJO"；实际项目没有单元测试但模型写"所有 Service 必须写单元测试"。

**验证方式**：随机抽检 2-3 条规范，去代码中验证是否确实成立。

**GREEN 规则**：SKILL 要求"从代码实际推断，不臆造"，置信度分级（高=多处一致证据；中=多数一致但有例外；低=仅个别样例），低置信度规范必须标注"仅个别样例，需进一步确认"。

### 失败模式 3：维度遗漏，覆盖不全

模型只提了命名规范和目录规范，漏掉错误处理规范、Git 提交规范、测试规范、封装规范等重要维度。

**验证方式**：检查产出是否覆盖了 skill 要求的全部 7 个维度（目录规范、命名规范、封装规范、前端样式规范、错误处理、测试规范、Git 提交规范）。

**GREEN 规则**：SKILL 分析策略定义了 7 个规范维度，每个维度必须有章节，不适用者需明确注明"未发现/不适用"而非省略。

### 失败模式 4：不适用维度强行凑数

项目没有前端样式规范（如纯后端项目）或没有测试文件，但模型仍然编造了"CSS 方案"或"测试覆盖率要求"等规范。

**验证方式**：检查不适用维度是否明确标注"未发现/不适用"，而非编造内容。

**GREEN 规则**：SKILL"边界处理"章节要求"某些规范维度不适用时标注'未发现/不适用'，不强制凑数"，极小项目豁免规则明确。

### 失败模式 5：无置信度标注，无法判断规范可靠度

模型列出一堆规范，但没有任何标注说明哪些是高置信度（多处证据一致）、哪些是低置信度（仅个别样例），使用者无法判断哪些规范值得信赖。

**验证方式**：检查每条规范是否附有"推断置信度"（高/中/低）字段。

**GREEN 规则**：SKILL 产出模板要求每条规范必须包含"推断置信度"字段，置信度分三级定义明确标准。

## 测试靶子：dsp 项目

- 路径：`/Users/sunlc/sunlc_work/sunlc_skills/dsp`
- 预期覆盖维度：目录规范（Maven 多模块 + Vue 前端）、命名规范（Java 类名/PascalCase + Vue 文件名）、封装规范（SPI 解耦 + 六层模块化）、前端样式规范（Element Plus + 无自定义 CSS 方案）、错误处理（BusinessException + ErrorCode + GlobalExceptionHandler）、测试规范（仅 1 个测试类）、Git 提交规范（CONTRIBUTING.md 中有约定）
- 预期低置信度维度：测试规范（仅有个别测试类，无法推断完整规范）、前端样式规范（可能无自定义 CSS 方案）
- 预期不适用维度：极小项目豁免不适用

## Baseline 实际跑偏记录

执行环境：dispatch 一个无 skill 约束的 baseline subagent，要求"直接分析 dsp 项目，提取开发规范"，禁止读 skills 目录、禁止读 docs/project-knowledge（避免前置产出污染），允许读 CLAUDE.md/CONTRIBUTING.md/源码。

**重要前提澄清**（baseline 执行后纠正了压力场景的几个假设）：
- dsp **有** `CONTRIBUTING.md`（显式含分支策略/提交规范/版本管理）和 `CLAUDE.md`（显式含行为准则）——所以"Git 提交规范"维度在 dsp 上是**显式文档驱动**的，不需要从 git log 反推。事实上 dsp 嵌在 skills 工作区里、无独立 git 历史，git log 无法反推。
- dsp **有** `dsp-admin-web/src/style.css`（完整 CSS 设计令牌系统，6.4 KB）——所以前端样式规范维度在 dsp 上**有实质内容**，不是"未发现"。
- dsp 实际有 **4 个测试类**（`SysUserServiceTest`/`XmlEngineTest`/`ResultMapperTest`/`JwtUtilTest`），分布在 4 个模块——测试规范维度**有少量样本但不足以外推完整规范**。

对照 5 个失败模式的实际跑偏：

### 实际跑偏 1：低置信度规范未标注置信度，使用者无法判断可靠度（对应失败模式 5，最严重）

baseline 产出**完全没有置信度字段**。所有规范平铺直叙，无法区分：
- 高置信来源（CONTRIBUTING.md 显式声明的分支/提交规范、CLAUDE.md 行为准则）
- 高置信来源（多处源码一致的 ApiRequest/ApiResponse 信封、ErrorCode 枚举）
- 低置信来源（仅 1 个 `XmlEngineTest` 样例外推的"测试方法名用 `场景_预期`"）

第 7.2 节「测试编写要点」从单一测试样例外推"测试方法名用描述性小驼峰：`场景_预期`"——这是典型的低置信度规范，但 baseline 把它和"所有对外接口必须走 ApiResponse 包装"（高置信）以同等语气陈述，使用者误以为两者可信度相同。

**结论**：失败模式 5 在 baseline 上**完全命中**。SKILL 必须强制每条规范附置信度（高/中/低）+ 证据，低置信度规范必须显式标注"仅个别样例，需进一步确认"。

### 实际跑偏 2：测试维度过度外推（对应失败模式 4，轻度凑数）

dsp 只有 4 个测试类（且分布在 4 个不同模块，无统一测试基类/抽象），但 baseline 第 7 章写了 3 个子章节（现状/编写要点/运行），其中 7.2「测试编写要点」给出"优先用模拟逻辑+断言验证纯函数行为，而非启动完整 Spring 上下文"——这是从 `XmlEngineTest` 单一样例的注释外推的"规范"。4 个测试类不足以支撑"测试规范"，正确做法是标注"项目仅 4 个测试类，测试规范未成型，此处仅记录现有样例的做法，置信度低"。

**结论**：失败模式 4 在 baseline 上**轻度命中**。SKILL 必须强制"样本不足时降级为'记录现有样例'+ 低置信度标注"，而非外推为规范。

### 实际跑偏 3：部分规范缺精确代码证据，依赖口头转述（对应失败模式 1，轻度）

baseline 大部分规范有文件级证据（如 `dsp-common/.../common/model/ApiRequest.java`），但部分规范只给了类名/方法名，没有代码行/片段：
- 第 4.2 节"HTTP 状态码映射约定"（4001/4002→401、4003→403 等）只说"证据：GlobalExceptionHandler.java"，没给出具体哪个 case 映射哪个状态码的代码行——读者要自己去翻源码验证。
- 第 5.3 节"动态数据源切换后必须在 finally 中 poll()"只给了方法名 `SqlExecutor.query()`，没给代码片段。

**结论**：失败模式 1 在 baseline 上**轻度命中**（baseline 整体有证据意识，但证据颗粒度不统一）。SKILL 必须明确证据的最小要求：文件路径 + 关键代码片段/模式描述，不能只给文件名。

### 实际跑偏 4：维度覆盖反而超额，但来源混淆（结构性问题，非预设失败模式）

baseline 覆盖了 10 个维度（目录/命名/封装/错误处理/安全/前端样式/测试/配置/Git/行为准则），超出 spec 要求的 7 维度。但 baseline **没有区分"规范来源类型"**：
- 显式文档来源（CONTRIBUTING.md、CLAUDE.md）——这是"项目声明的规范"
- 代码推断来源（ApiRequest 信封、ErrorCode 枚举、设计令牌）——这是"代码事实上的规范"

这两类混在一起平铺，使用者无法知道"这条规范是项目明文要求，还是我从代码反推的"。例如 Git 提交规范是 CONTRIBUTING.md 明文，而"Controller 不写业务逻辑"是从代码反推的，但 baseline 用同样的语气陈述。

**结论**：这是 baseline 暴露的**新跑偏点**（不在原 5 个失败模式内）。GREEN 规则需补充：每条规范应标注来源类型（显式文档 / 代码推断 / 混合），与置信度字段并列。

### 实际跑偏 5：维度遗漏未发生（对应失败模式 3，未命中）

baseline 因为读到了 CONTRIBUTING.md/CLAUDE.md，**没有遗漏任何 spec 要求的 7 维度**。但这是 baseline 运气好（dsp 恰好有显式文档）。**真正风险**：若目标项目**没有** CONTRIBUTING.md/CLAUDE.md（很多项目确实没有），baseline 极可能遗漏 Git 提交规范、行为准则维度——因为这些维度很难纯从代码反推。

**结论**：失败模式 3 在 dsp baseline 上**未命中**，但 SKILL 仍需强制 7 维度全覆盖（不适用者注明"未发现/不适用"），因为这是"防项目无显式文档时遗漏"的保险。

### Baseline 总结

- 命中失败模式：5（置信度缺失，最严重）、4（测试过度外推）、1（证据颗粒度不统一）轻度
- 新增跑偏点：来源类型混淆（显式文档 vs 代码推断未区分）
- 未命中失败模式：2（规范与真实用法不符，baseline 抽样较扎实）、3（维度遗漏，因 dsp 有显式文档）
- SKILL 需额外编码的规则：每条规范必须含【证据（文件+片段）】【置信度（高/中/低）】【来源类型（显式文档/代码推断/混合）】三字段
