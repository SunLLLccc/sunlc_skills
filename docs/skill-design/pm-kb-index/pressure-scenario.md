# pm-kb-index 压力场景

## Baseline 失败模式

不加 pm-kb-index skill，让模型直接"为 KB 文档汇编总索引 README"时，预期出现以下 5 类跑偏：

### 失败模式 1：平铺罗列文档清单，无导航价值

模型把知识库里的所有文档列成一张表或一个目录列表（"01-项目概览.md / 02-技术栈与架构.md / ..."），仅给出文件名，没有"这份文档讲什么、什么时候读它"的导航信息。使用者打开 README 看到的只是一个目录树，与直接 `ls` 没有区别，没有降低 KB 的查阅成本。

**验证方式**：检查 README 是否对**每份文档**都给了"何时读哪份"的一句话说明（不是仅给文件名/标题），使用者据此就能判断"我现在要做 X，该打开哪几份"。

**GREEN 规则**：SKILL 要求每份文档的导航条目必须含「何时读」一句话，且 README 的核心价值是"任务驱动的查阅入口"而非"目录树"。产出模板含每份文档的"何时读"字段。

### 失败模式 2：无基于任务的阅读路径

模型只给"文档导航"（每份文档一句话），但不给"任务驱动的阅读路径"——使用者面对"我要开发一个新功能"、"我要部署到生产"、"我要改一个 API"、"我要排查一个 bug"等具体任务时，README 不能回答"按什么顺序读哪几份"。读者必须自己读完所有文档才能拼凑出路径。

**验证方式**：检查 README 是否含 ≥2 条**基于任务的阅读路径**（如"要开发某功能→先读 02+03+04"、"要部署→读 05"、"要改 API→读 04+02"），每条路径明确"先读什么、再读什么"的顺序。

**GREEN 规则**：SKILL 要求 README 含"基于任务的阅读路径"章节，路径数 ≥2 条，每条标注任务名 + 阅读顺序（含跳读哪些章节的提示），且覆盖项目最典型的工作流（开发 / 部署 / 改 API 至少各一条）。

### 失败模式 3：元信息缺失或臆造

模型不写元信息（生成时间、源码版本/commit、KB 文档清单与状态），或臆造生成时间（写一个当前时间）而不查 `_meta/` 目录。元信息缺失导致使用者无法判断"这份 KB 是基于哪个版本生成的、是否过期、是否完整"，也无法回答"KB 是否已覆盖全部规划文档"。

**验证方式**：
- 检查 README 是否含"元信息"章节（生成时间、源码版本/commit、KB 文档清单与状态）。
- 检查生成时间是否来自 `_meta/` 目录的 `scannedAt` 等字段（若存在），或当 `_meta/` 不含时间戳时是否**显式标注"未生成/未记录"**而非编造一个时间。

**GREEN 规则**：SKILL 要求元信息从 `_meta/` 读取（`project-type.json` 的 `scannedAt` / `scanVersion` / `projectName`，`manifest.json` 若存在则读更多），**元信息字段无法读取时显式标注"未生成/未记录"，禁止臆造**。

### 失败模式 4：缺失文档未标注

知识库的规划文档可能并未全部生成（如计划有 01-06，实际只有 01-05；或 06-校验报告因波次 3 未跑而未生成）。模型要么只列出已生成的文档（让使用者误以为 KB 就这些），要么把未生成的也列上但链接成死链（点开 404），不告知使用者"这份还没生成"。

**验证方式**：检查 README 是否对**规划中但未生成**的文档显式标注"未生成"，且不给出死链；对已生成的文档给出可用链接。

**GREEN 规则**：SKILL 要求 README 区分"已生成"与"未生成"两类文档，未生成的文档在导航中明确标注"未生成"且不提供链接（或链接前注明"待生成"），不静默忽略也不制造死链。

### 失败模式 5：导航无"何时读"，只有"是什么"

即使模型给出了文档导航（不是失败模式 1 的纯目录树），每份文档的描述也常写成"这份文档讲了 X、Y、Z"（陈述内容），而不是"当你想 Y 时读这份"（触发条件）。两者差别巨大：内容描述让读者"先读才知道有没有用"，触发描述让读者"看一眼就知道要不要读"。

**验证方式**：抽查每份文档的导航条目，检查描述是否以"何时读/想 Y 时读"的触发式表述为主（而非仅"这份文档讲 X"的内容式表述）。

**GREEN 规则**：SKILL 要求每份文档的导航描述采用**触发式**（"当你想 X 时读这份 / 想做 Y 时先读这份"），而非纯内容陈述。模板示例体现这一点。

## Baseline 实际跑偏记录

执行环境：dispatch 一个无 skill 约束的 baseline subagent，要求"为 dsp/docs/project-knowledge/ 下的 KB 文档汇编总索引 README，不要参考任何 skill"，禁止读 skills 目录、禁止读 docs/skill-design 目录。baseline 整体质量**高于预期**——它有项目一句话简介、文档地图表、任务驱动阅读路径，但对照 5 个失败模式仍有明确命中点。

### 实际跑偏 1：元信息维度整体缺失（对应失败模式 3，最严重且最普遍）

baseline 产出的 7 章 README 中，**完全没有元信息章节**——没有生成时间、没有源码版本/commit、没有 KB 文档清单与状态声明、没有 skill 版本信息。`_meta/project-type.json` 里明确有 `scannedAt: "2026-06-14T02:30:00+08:00"` 和 `scanVersion: "pm-scan v1"`，baseline 完全没用。读者无法判断"这份 KB 基于源码的哪个版本生成、是否过期、是否完整"。

这是失败模式 3 的典型表现：**整个元信息维度被忽略**。比预测的"编造当前时间"更糟——baseline 连编造都没做，直接跳过了。

**结论**：失败模式 3 命中（最严重）。GREEN 规则：README 必须含元信息章节，从 `_meta/project-type.json` 读 `scannedAt`/`scanVersion`/`projectName`，从 `_meta/manifest.json`（若存在）读更多；**字段无法读取时显式标注"未生成/未记录"，禁止编造当前时间、禁止跳过该章节**。

### 实际跑偏 2：缺失文档（06-校验报告）完全未提及（对应失败模式 4，命中）

baseline 列了 01-05 共 5 份文档，**完全没提"06-校验报告"**。波次 3 的校验报告（pm-verify 产出）尚未生成，但使用者看 README 会以为 KB 就这 5 份，不知道"还有规划中的 06 待补、KB 当前是不完整的"。这是静默忽略——既不标"未生成"，也不告知使用者"KB 文档清单的完整规划"。

**结论**：失败模式 4 命中。GREEN 规则：README 必须区分"已生成"与"未生成"文档，未生成的（如 06-校验报告）在导航中显式标注"未生成"，不静默忽略、不给死链，让使用者知道 KB 的完整规划与当前完成度。

### 实际跑偏 3：文档导航多为内容陈述，缺触发式"何时读"（对应失败模式 5，部分命中）

baseline 的"知识库文档地图"表格用了三列："一句话定位 / 回答什么问题 / 核心内容"——这三列都是**内容式表述**（"项目地形图 / 这是什么项目、多大、入口在哪 / 目录结构、规模指标、入口点"）。读者看了知道"文档里有这些内容"，但不够明确"我什么时候要打开它"。不过 baseline 的"按任务找文档"章节本身是强触发式的（"我刚接手 / 我要把项目跑起来 / 我要改一个 API"），所以整体有任务驱动思维，只是导航表的描述风格偏内容陈述。

**结论**：失败模式 5 部分命中（任务路径章节做对了，但文档导航表的描述风格偏内容陈述）。GREEN 规则：文档导航表必须含"何时读"列，采用触发式表述（"当你想 X 时读这份 / 想做 Y 时先读这份"），与内容陈述区分。

### 实际跑偏 4：无任务型阅读路径？——未命中（对应失败模式 2，未命中）

预测的失败模式 2（无基于任务的阅读路径）**未命中**。baseline 有完整的"按任务找文档"章节，含 8 条任务路径（刚接手 / 跑起来 / 改后端 API / 改前端 / 改引擎 / 接新数据源 / 排查报错 / 部署 / 提 PR），每条标注阅读顺序与红线提示，质量很高。这说明 baseline 的 LLM 自身有较强的"任务驱动导航"意识。

**结论**：失败模式 2 未命中。但 SKILL 仍应编码"任务型阅读路径"规则——因为这是 README 的核心价值，baseline 做对可能是运气/能力波动，规则化能保证一致性。GREEN 规则保留：README 含"基于任务的阅读路径"章节，≥2 条。

### 实际跑偏 5：平铺无导航？——未命中（对应失败模式 1，未命中）

预测的失败模式 1（平铺无导航）**未命中**。baseline 有项目一句话简介（"DSP 是一个前后端分离 + 微服务部署的 Maven 多模块 monorepo，核心是用 XML 配置零代码接入数据接口"），有文档地图表，有任务章节，定位明确是"知识库前门"而非"目录树"。

**结论**：失败模式 1 未命中。但 SKILL 仍应编码"项目一句话简介开篇"规则，保证 README 的"前门"定位。

### Baseline 总结

- **明确命中失败模式**：3（元信息维度整体缺失，最严重）、4（缺文档 06 未标注）
- **部分命中失败模式**：5（文档导航表偏内容陈述，但任务章节做对了）
- **未命中失败模式**：1（平铺无导航）、2（无任务型阅读路径）—— baseline 在这两点上质量高于预期
- **整体观察**：baseline 的 README **导航与任务驱动做得好**（项目简介、文档地图、任务路径齐全），但**元信息维度完全缺失**、**缺文档未标注**。这两个是 baseline 的真实短板，也是 SKILL 必须强制编码的规则。SKILL 的核心价值在于：
  1. 强制元信息章节存在（从 `_meta/` 读，缺失标注，禁止跳过）
  2. 强制区分已生成/未生成文档（缺文档标注"未生成"）
  3. 文档导航采用触发式"何时读"表述（baseline 做对了任务章节，但导航表风格需规范化）
  4. 任务型阅读路径 ≥2 条（baseline 做对了，规则化保证一致性）
  5. 项目一句话简介开篇（baseline 做对了，规则化保证一致性）
