# lp-feature-scan 压力场景（TDD）

## 失败模式

扫描项目识别"可学习功能"时常见的失败：

1. **通用模板化**：把功能识别成"用户登录/CRUD/权限管理"这类业界通用清单，而非该项目特有的、值得学习的能力单元。
2. **粒度失控**：过细（每个 API 端点 / 每个文件一个"功能"）或过粗（"整个后端"算一个功能）。
3. **字段缺失**：清单只有功能名，缺核心度/复杂度/依赖/证据，导致后续无法排序、无法深挖、无法生成提示词。
4. **证据不可验证**：证据写成"相关代码"等模糊描述，或指向不存在/不相关的路径。
5. **不可挑选**：清单是纯叙述段落，没有勾选框，无法支持人在回路挑选。

## 验证方式

在目标项目（验证靶子）上跑 lp-feature-scan，产出 `features/inventory.md`，逐条核对：

- [ ] 每条是**项目特定**的功能（非通用模板），名称是项目语境下的自然名称。
- [ ] 每条含四字段：核心度（核心/边缘）、复杂度（高/中/低）、依赖（学它前先懂啥 / 无）、证据（真实路径 + 说明）。
- [ ] 证据指向的文件/目录**真实存在**且确属该功能核心。
- [ ] 清单**按核心度排序**（核心在前）。
- [ ] 每条带 `- [ ]` 勾选框，供人在回路标记 `- [x]`。
- [ ] 零臆造（没有项目里不存在的功能/路径）。

## GREEN 规则

inventory.md 至少识别出靶子项目的多个**项目特定**功能（涵盖核心业务能力与项目自封装/特色能力，具体由执行者据实识别，不在本场景点名），每条四字段齐全、证据真实可验证、带勾选框、按核心度排序、零臆造。满足即 GREEN。
