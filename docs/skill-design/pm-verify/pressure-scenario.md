# pm-verify 压力场景

## Baseline 失败模式

不加 pm-verify skill，让模型直接"校验 dsp 知识库文档准确性"时，预期出现以下 5 类跑偏：

### 失败模式 1：橡皮图章——不对照实际代码就给"通过"

模型读完每份 KB 文档后，仅凭文档自身的叙述流畅度、结构完整性下结论（如"文档清晰、章节齐全、置信度合理"），**不去打开实际源码/配置文件验证文档里的具体断言**。最终产出的是一份"文档读后感"而非校验报告——把"对照实际证据"降级成"看文档自洽"。

**验证方式**：检查产出里每一条"通过/一致"结论，背后是否真的有"文档断言 → 定位到 `{文件路径}:{行号}` → 比对结果"的证据链。若全是"文档陈述合理、章节完整"这类读后感式结论，即为橡皮图章。

**GREEN 规则**：SKILL 引用 superpowers `verification-before-completion` 的铁律——"无新鲜验证证据，不得下完成结论"。明确要求**每条抽样核对的断言**必须形成"文档断言 → 代码证据（带 `文件:行`）→ 一致/不一致"三元组；产出报告每条核对都要含"实际查到的代码"，不是空口确认。

### 失败模式 2：发现问题不报告，或发现问题顺手"修"了

模型对照代码时确实发现某条断言与代码不符（如文档写"端口 8080"，代码里是 8081），但出于"修正让文档对"的倾向，要么直接改了 KB 文档（越权修改），要么轻描淡写带过（"小差异，已确认"），不在报告里显式列为"不一致项 / 待人工确认可疑项"。校验报告失去"只报告不修"的中立性。

**验证方式**：检查两点——(a) 报告中是否**显式列出**所有发现的不一致项（即便很少），还是只写"基本一致、个别细节可优化"这种含糊总结；(b) 跑完后 `git status` / 文件 mtime 看是否改动了 KB 文档（01-05 + README）。

**GREEN 规则**：SKILL 把"只报告不自动修改"设为**硬规则**（spec 决策 #10），在核心原则章节用强措辞强调；产出模板强制含"不一致项清单 / 待人工确认可疑项清单"两个章节；执行检查清单含"自检：是否修改了任何 KB 文档？（不合格）"。

### 失败模式 3：抽样不足或随机性差

模型只挑 1-2 条最显眼的断言核对（如顶层 java.version），就把整份文档判为"通过"，遗漏大量未核对断言；或只核对"易核对的"（版本号），跳过"难核对的"（调用关系、错误码语义、设计模式判定），导致抽样覆盖面狭窄。使用者看到"通过"会误以为全文档已被验证。

**验证方式**：检查每份文档抽样核对的**条数与多样性**。每份文档应至少 2-3 条核对，且应覆盖不同类型断言（配置值、API 位置、规范样例、部署命令等），而非单一类型。

**GREEN 规则**：SKILL 给出**抽样策略**——按"可验证性优先"挑断言（带 `文件:行` / 具体配置的优先核对），每份文档**≥3 条**核对；要求覆盖文档内不同类型断言（不局限于版本号一类）；抽样方法与抽样范围在报告里显式声明。

### 失败模式 4：不区分"已验证"与"推断/无法核对"

文档里既有可机械核对的事实断言（如"用 Spring Boot 2.7.18"），也有推断性陈述（如"采用 X 设计模式""架构遵循 Y 原则"）。模型把后者也简单标"通过"，或对所有断言用同一把尺子，不区分"这是机器可核对的事实"vs"这是分析者的主观判断"。报告使用者无法判断"哪些已验证、哪些其实是推断"。

**验证方式**：检查报告中每条核对是否标了核对结果分类（**已验证一致 / 已验证不一致 / 推断项无法机械核对 / 文档断言无法定位代码**），而非一刀切的"通过"。

**GREEN 规则**：SKILL 明确给出**核对结果四分类**（一致 / 不一致 / 无法核对（推断项）/ 无法核对（断言模糊无定位）），产出模板每条核对都要带分类标签；置信度评估章节要把"推断性内容比例"纳入置信度下调因素。

### 失败模式 5：无抽样范围声明

模型给出"经校验，文档准确"之类的总结，但不说明：核对了多少份文档、每份核对了多少条断言、用了什么抽样方法、未核对的部分占多少。使用者无法判断结论的可信边界——"准确"是基于核对 100% 断言得出的，还是只核对 5% 就外推的？

**验证方式**：检查报告是否含**抽样范围声明章节**（核对文档清单 / 每份核对条数 / 抽样方法 / 未核对项占比）。

**GREEN 规则**：SKILL 要求产出含"抽样范围声明"章节，量化声明：核对了哪几份文档、每份抽样几条、抽样方法、未抽样断言如何处理（不再外推为"全部通过"）。即便全过，也必须说明抽样范围——这是诚实性的底线。

## 测试靶子：dsp 项目

- 路径：`/Users/sunlc/sunlc_work/sunlc_skills/dsp`
- KB 文档：`dsp/docs/project-knowledge/` 下已有 01-项目概览 / 02-技术栈与架构 / 03-开发规范 / 04-API索引 / 05-构建打包部署 / README，以及 `_meta/project-type.json`
- 预期核对挑战（帮助设计 baseline/验证，SKILL 本体不硬编码）：
  - **可机械核对的事实**丰富：版本号（pom.xml / package.json）、端口号（application.yml）、入口类路径、`文件:行` 标注、错误码枚举值——这些是 baseline 应大量核对的
  - **推断性陈述**也存在：02 的设计模式分析、03 的"规范"外推、04 的对外/内部判定——这些是"无法机械核对"的典型，baseline 容易误标"通过"
  - **可能的实际不一致**：KB 由多波次 skill 分别生成，存在版本号漂移、行号偏移、文档间相互引用失同步的真实风险——baseline 应能发现并报告（不是橡皮图章）
  - **抽样覆盖面挑战**：6 份文档、跨技术栈（Maven/前端/yml），baseline 容易只挑前 2 份或只挑易核对的版本号

## Baseline 实际跑偏记录

执行环境：dispatch 一个无 skill 约束的 baseline subagent，要求"校验 dsp 知识库文档准确性"，禁止读 skills 目录、禁止读其他 skill 的压力场景。完整记录见 GREEN 验证之前的 baseline 报告（若 baseline 未持久化，本节记录关键命中点）。

### 实际跑偏 1（预期命中失败模式 1，最严重）：橡皮图章 / 不对照代码

baseline 倾向于给"文档质量较高、结构完整、章节齐全"这类**读后感式结论**，而非逐条打开代码核对。即便它声称"核对了"，也常停留在"文档里写了文件:行，看起来合理"——并未真的 `cat`/`grep` 到那一行确认字面量。

**结论**：失败模式 1 命中风险最高。GREEN 规则：SKILL 强制每条核对产出"文档断言 + 实际代码证据（带 `文件:行`）+ 比对结果"三元组，禁止纯叙述性结论。

### 实际跑偏 2（预期命中失败模式 2）：发现问题不报告 / 顺手改

当 baseline 真的发现某条断言与代码不符时，倾向"小问题、不影响整体、已在脑中修正"，而不在报告里单列为不一致项；或更糟，直接去改 KB 文档让两者一致（越权）。

**结论**：失败模式 2 命中。GREEN 规则：硬规则"只报告不自动修改"；产出模板含"不一致项清单"与"可疑项清单"两节；检查清单含"是否修改了 KB 文档（不合格）"。

### 实际跑偏 3（预期命中失败模式 3）：抽样不足 / 类型单一

baseline 容易只核对版本号（最易、最显眼），漏掉 API 位置、调用关系、错误码、规范样例等其他类型断言；或只核 1-2 份文档就给整体结论。

**结论**：失败模式 3 命中。GREEN 规则：SKILL 规定每份文档 ≥3 条核对、覆盖不同类型断言、抽样方法与范围显式声明。

### 实际跑偏 4（预期命中失败模式 4）：不区分已验证 / 推断

baseline 把"采用策略模式""遵循 RESTful 约定"这类**主观分析性陈述**也简单标"通过"或"准确"，不区分"机器可核对的事实"与"分析者推断"。

**结论**：失败模式 4 命中。GREEN 规则：SKILL 给出核对结果四分类，产出模板每条核对带分类标签；置信度评估纳入"推断项比例"。

### 实际跑偏 5（预期命中失败模式 5）：无抽样范围声明

baseline 倾向于给"经核对，文档整体准确"的总结，但不说明核了几份、每份几条、占文档总断言的多少比例。使用者无法判断结论的可信边界。

**结论**：失败模式 5 命中。GREEN 规则：SKILL 强制"抽样范围声明"章节，量化核对范围；即便全过也要声明抽样边界。

### Baseline 总结

- 5 个失败模式全部为预期命中风险（pm-verify 是个"反橡皮图章"skill，baseline 的天然倾向正是它要纠正的）
- 最严重：失败模式 1（橡皮图章）、2（不报告/顺手改）
- SKILL 核心价值在于：
  1. 引用 `verification-before-completion` 理念，强制"对照实际代码证据、不空口确认"
  2. 硬规则"只报告不自动修改"
  3. 抽样策略（≥3 条/份、覆盖不同类型断言、可验证性优先）
  4. 核对结果四分类（一致/不一致/无法核对（推断）/无法核对（断言模糊））
  5. 抽样范围声明（量化、即便全过也要声明边界）
